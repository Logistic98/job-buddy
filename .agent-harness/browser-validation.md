# Browser Validation Playbook

本文件定义 `job-buddy` 中前端和用户可见交互改动的浏览器验证要求。凡涉及工作台交互、登录/扫码弹窗、SSE 流程、岗位卡片、原岗位预览、会话恢复、状态管理或用户可见 UI 行为，都必须执行本验证，而不能只依赖单元测试、接口测试或 `npm run build`。

## 风控安全红线（先读后验）

本手册是整个自检体系中唯一可能产生真实 Boss 直聘流量的环节。自动化脚本（`doctor.sh`、`verify.sh`、`evaluate.sh`、`gate.sh`）与单元测试均不触达 Boss；浏览器验证若触发登录状态导入、岗位搜索或岗位详情访问，必须严格遵守以下约束，避免自检本身把账号搞进风控：

- 复用 PostgreSQL `auth_state` 中的持久化登录态，不要反复扫码登录；凭证仅在工具调用期间注入 agent-tool 内存。
- 浏览器 Cookie 导入默认关闭；只有明确接受本机钥匙串授权时才能显式开启。
- 真实 Boss 动作只跑最小一遍。同一项验证不要反复发起搜索或重复打开岗位详情；一次能确认 UI 行为即可，不要为“多看几眼”重复触发深链访问。
- 尊重 Boss 工具限速与配额，不要绕过 RateLimiter。出现限速、冷却、硬停提示属预期保护，不允许通过改配置、清状态文件或直连绕开。
- 只有改动确实落在 Boss 实时链路（搜索、详情、扫码、原岗位预览）时才驱动真实 Boss；纯前端、SSE、状态管理类改动优先用 Mock / 既有 Fixture / 非 Boss 路径验证。
- 一旦出现验证码、滑动验证、“访问异常/环境异常/操作过于频繁”等风控信号，立即停手，不得重试或加速，按冷却时间等待后再人工判断。
- 不要在短时间内串行跑多轮完整 E2E。多人或多分支验证同一账号时需错峰，避免叠加流量。

## 启动服务

先启动根目录 `.env` 已配置的本地服务链。统一启动脚本按当前依赖顺序启动 Sandbox、Backend、Memory、Tool、Runtime、Intent、Eval 和 Frontend，避免本手册重复维护端口与启动参数：

```bash
./scripts/start-all.sh
```

脚本会输出实际访问地址、日志目录和 PID 目录。默认前端地址为：

```text
http://localhost:5173
```

## 必查项

### 通用前端交互

- 页面可以正常打开，无白屏。
- 控制台没有阻断交互的 error。
- 表单输入、按钮点击、弹窗打开/关闭符合预期。
- 加载态会结束，不会无限 pending。
- 错误提示可见、可理解，不会被静默吞掉。
- 修改过的状态在切换会话、刷新页面后仍符合预期。

### SSE / 工作台流程

- 发起请求后，思考与工具执行过程可以展开查看。
- 执行过程中该过程面板不会自动折叠导致用户看不到进度。
- `done` 后 UI 不再显示“仍在执行”。
- 如果需求要求“结果出来即停止”，必须确认后端不再发送额外结论或二次结果。
- 历史消息重新打开时，工具过程、岗位卡片、匹配结果等关键上下文不丢失。

### Boss 直聘登录与岗位推荐

- 先调用 `/api/boss/login-status` 或等价流程确认登录态，不允许未确认就弹扫码。
- 如果登录态有效，工作台直接继续执行，不弹二维码。
- 如果登录态无效，才显示 Boss 扫码弹窗，并且二维码图片可见；但 HTTP 二维码登录可能缺少 `__zp_stoken__`，若扫码后仍无效，应在常用浏览器中登录 Boss 后重新导入 Cookie。
- 扫码完成后应从当前任务检查点继续，不应重复执行完整前置流程。
- 验证 `/api/boss/login-status` 能从 PostgreSQL `auth_state` 恢复状态，agent-tool 未生成 `credential.json` 或持久浏览器 profile，Redis 风控键均设置 TTL。

- 岗位列表默认不带职位描述（JD），需点击岗位卡片上的“加载职位描述”按钮才懒加载 JD；未登录时应触发扫码弹窗而非静默失败。
- 原岗位预览/打开如需要登录态，应复用后端注入 agent-tool 内存的 Cookie。

## 建议使用浏览器工具记录证据

验证时建议记录：

1. 访问地址。
2. 实际用户路径。
3. 关键页面截图或 DOM 观察结果。
4. 网络请求中关键接口状态。
5. 控制台错误检查结果。
6. 未覆盖项及原因。

示例记录：

```text
访问地址：http://localhost:5173
用户路径：登录工作台 -> 输入岗位推荐请求 -> 等待 Boss 登录状态检查 -> 展示岗位卡片
观察结果：登录态检查完成后未弹二维码；思考与工具执行过程保持展开；岗位卡片出现后 SSE 结束，无额外推荐结论。
未覆盖项：未实际搜索，原因是当前账号处于冷却期。
```

## 失败处理

如果浏览器验证失败，必须先修复交互问题再声明完成。禁止只以测试通过、构建通过、接口返回正常作为前端交互任务的完成依据。
