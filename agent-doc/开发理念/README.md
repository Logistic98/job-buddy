# 开发理念

本目录沉淀 job-buddy 仓库的 AI 协作开发理念，是 CLAUDE.md 中"Agent 开发理念"章节的展开说明。CLAUDE.md 保持精简摘要，本目录承载完整论述；两者不一致时，应先修订本目录文档，再同步 CLAUDE.md。

## 文档即代码

在开始编码之前，先把架构、模块设计、库表、接口等文档写清楚，作为 AI 生成代码的输入。文档不是交付后的附属物，而是与代码同等地位的工程资产：AI 能否稳定产出符合预期的代码，直接取决于输入文档的完整性与准确性。

- 关键改动必须先在 `agent-doc/开发文档/` 落方案文档，内容覆盖：为什么做、方案是什么、具体怎么做、涉及哪些模块和接口、风险、验证方式、后续演进。
- 文档随实现持续维护，禁止文档与代码长期不一致；实现偏离方案时要么改实现，要么改文档并说明理由。
- 关键设计决策记录在仓库内，作为未来 AI 与人共同的上下文，避免决策依据只存在于对话或个人记忆中。

## 规范驱动开发

每次让 AI 生成代码前，先用自然语言把目标、边界、约束、禁止事项说清楚。规范是 AI 的"类型系统"：规范越明确，产出偏差越小。

- 任务粒度尽量小、闭环明确，避免一次性让 AI 改动跨多个模块的大范围代码。
- 明确"不要做什么"与"要做什么"同等重要：禁止顺手重构、禁止引入新框架、禁止硬编码等约束必须显式声明。
- 每实现若干功能模块后，安排一次重构，清理冗余、统一风格，防止技术债积累。

## 测试驱动开发

关键模块的测试用例优先于实现，由 AI 根据规范生成首版，人工校验后作为代码生成的契约。

- 用例覆盖主流程、边界条件与典型异常，不允许只覆盖 happy path。
- 测试通过只是最低门槛，人工评审仍需关注代码规范、日志完整性、并发安全与资源管理。
- 测试是 Agent 自我验证的核心手段：没有测试的模块，AI 无法在循环中确认自己的改动是否正确。

## 可验证目标

给 AI 的任务必须能在对话或日志中被验证。不可验证的目标会让 AI 的"完成声明"失去意义。

- 目标示例："`uv run pytest tests/test_x.py` 全部通过"、"`/health` 返回 200"、"gate.sh agent-runtime --quick 通过"。
- 模糊目标（如"性能更好"、"代码更优雅"）必须先转化为可执行命令或可对比指标。
- 长任务必须设定停止条件：最大轮次、最长运行时间、最大 diff 范围、允许修改的目录。

## Harness 优先

优先投入测试、lint、类型检查、可复现的本地运行环境，让 Agent 可以自我验证。Harness 是 Agent 工程的地基，功能是地基之上的建筑。

- CI、Dockerfile、启动脚本、健康检查、监控告警等基础设施在功能开发之前先打通。
- 本仓库的 Harness 落地为 `.agent-harness/`：`verify.sh` 负责测试/构建，`evaluate.sh` 负责行为评估，`gate.sh` 是交付统一门禁，`doctor.sh` 负责环境自检。
- 自动化任务（Loop、夜间任务）必须显式设置预算与软着陆机制，失败时输出可读现场。
