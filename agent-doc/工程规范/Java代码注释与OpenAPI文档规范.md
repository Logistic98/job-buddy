# Java 代码注释与 OpenAPI 文档规范

## 目标与适用范围

Java 注释和 OpenAPI 元数据用于解释代码本身无法完整表达的职责、协议语义、边界条件与失败行为。注释必须与当前实现一致，不记录迭代过程、临时方案或未来计划。本文适用于 `agent-backend` 的 Controller、公开 DTO、Service 接口、基础设施适配器以及包含非直观约束的核心实现。

代码注释不追求覆盖每一行。类名、方法名和普通控制流已经能清楚表达含义时不重复解释；涉及租户隔离、权限委派、幂等、资源生命周期、外部依赖降级、数据格式兼容或安全限制时，应说明设计原因和不可突破的边界。公共接口使用 Javadoc 表达职责和契约，局部注释只解释紧邻代码的非直观决策。禁止使用“获取数据”“处理逻辑”“统一接口响应”一类不能增加信息量的模板注释。

## Controller 与 OpenAPI

每个 Controller 必须使用 `@Tag` 提供稳定、面向业务的中文分组名称；分组名称使用“领域能力 + 接口”，避免直接暴露 Java 类名。每个请求映射方法必须显式使用 `@Operation` 提供动宾结构的中文摘要，例如“查询工作区状态”“替换角色菜单”，不能依赖 SpringDoc 从方法名生成 `get`、`save`、`delete` 等无业务语义的标题。接口存在副作用、特殊空值语义、权限边界或非直观失败条件时，应同时填写 `description`。

路径参数和含义不明显的查询参数使用 `@Parameter` 说明语义，并提供不包含个人信息、密钥或真实业务数据的示例。请求体结构由 DTO 表达；请求体本身具有特殊语义时，再使用 OpenAPI `@RequestBody` 补充说明。SSE、文件上传下载和二进制预览接口必须准确声明媒体类型，并在说明中写明结束条件或内容类型。全局安全要求默认覆盖受保护接口；公开接口必须显式声明空安全要求，避免文档误导调用方。

## 请求与响应模型

跨 Controller 边界的请求和响应对象应使用明确 DTO，并通过 `@Schema` 描述模型职责和非自明字段。字段说明应包含有效格式、允许值、单位、默认行为或空值语义中的必要部分；示例必须稳定、最小且不敏感。枚举型字符串使用 `allowableValues`，时间字段明确格式，标识字段说明其所属资源。Swagger 元数据只描述既有校验和行为，不能声明实现中不存在的必填、长度或默认值约束。

所有普通 JSON 接口使用 `ApiResponse<T>` 统一表达 `code`、`message` 和 `data`。`code` 可以使用 HTTP 状态码或已登记的扩展业务码；当前 Boss 登录引导使用业务码 `4001` 和 HTTP 200，下游依赖失败使用业务码 `5001`。文档中的错误说明必须与全局异常处理器保持一致。直接返回 `SseEmitter`、文件流或其他非 JSON 类型的接口不套用统一响应结构，但必须准确描述传输协议。

## 验证与维护

新增或修改 Controller 后，至少运行 `mvn test`，并确认 `/v3/api-docs` 可以生成、Knife4j `/doc.html` 可以加载。`SpringBoot3ContractTest` 会检查后端 Controller 是否具有 `@Tag`、每个请求映射是否具有非空 `@Operation` 摘要，并验证关键 OpenAPI 元数据。新增接口不得通过修改测试白名单绕过文档要求。

OpenAPI 注解、Javadoc 和 `agent-backend/README.md` 必须随接口契约同步维护。仅修改注释和文档元数据时不得改动路由、HTTP 方法、请求响应结构、权限注解或业务调用顺序；交付前应审查差异，确认没有功能性变化。
