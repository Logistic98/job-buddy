# 对话主链路单流路由与理解往返消除方案

## 1. 背景与动机

当前工作台问答的首字延迟主要由两次串行的 Runtime 往返叠加而成。第一次是任务理解调用：`ChatSseServiceImpl.handle` 在进入流式合成之前，先以 `understanding_only=true`、`budget(1,0,1)` 的元数据向 Agent Runtime 发起一次非流式请求，仅为拿到意图、能力路由与 directive；这一调用即便已经按提供方关闭了思考链，实测仍需约两秒到两秒四。第二次才是真正的流式托管调用 `handleRuntimeManagedTask`，它再次携带个人上下文进入 Runtime 图，从 Task Understanding 一路走到流式合成，合成 prefill 又需约一秒。两次调用各自独立建连、各自重新装配上下文，理解阶段产出的中间状态无法传递给合成阶段，用户在看到第一个答案 token 之前必须串行等待“理解往返 + 合成 prefill”两段时间。

`工作台问答Token流式输出方案.md` 第九、第十节已经把首字耗时拆解清楚，并通过会话捷径、提示词瘦身、理解调用关闭思考链等手段把理解往返从约四秒压到两秒级。但只要理解仍是一次发生在流式合成之前的独立 LLM 往返，它就始终是首字关键路径上一段无法与合成重叠的串行等待。本方案的目标，是把“理解”和“合成”收敛进同一次流式请求，由 Runtime 在流内尽早抛出路由结果供 Java 决策，从关键路径上彻底移除这次独立往返，使常见的托管问答路径只保留一次 Runtime 流式请求。

## 2. 现状链路

现状的一轮托管问答按如下顺序串行执行。Java 侧 `handle` 先下发 `session` 与若干 `tool_status` 占位事件，再调用 `intentService.classify` 得到廉价预判并以 `intent_precheck` 透出。随后进入 `runTaskUnderstanding`，向 Runtime 发起非流式理解请求，从返回结果中抽取 directive 与意图，并以 `intent` 事件下发、据此走 `handleDirective` 的能力分发。分发分两类：一类是 Java 侧确定性能力，包括 Boss 登录、岗位推荐、简历匹配、简历分析，它们由 Java 自有服务执行，不需要 Runtime 流式合成；另一类是 Runtime 托管任务，进入 `handleRuntimeManagedTask`，携带由 `runtimeManagedMetadata` 装配的元数据（含基于已解析意图构建的 `personal_context`）发起第二次 Runtime 请求，这次是流式的，逐字回传 `message_delta` 与 `reasoning_delta`。

由此可见，对最常见的开放域问答与复杂工程问答而言，一轮交互要付出两次 Runtime 往返；个人上下文也被装配两次，一次在理解阶段以空意图装配，一次在合成阶段以已解析意图装配，二者内容不同且都进入模型提示词，无法简单合并。

## 3. 总体方案

### 3.1 方案选型

核心思路是把路由决策内嵌进流式请求：Runtime 在一次流式请求内先完成 Task Understanding，并在开始下发答案 token 之前，通过一个新的早期 SSE 事件把 directive 与意图抛给 Java；Java 据此事件完成能力分发判断。若判断结果是 Runtime 可独立承载的托管合成，Runtime 在同一请求内无缝继续流式合成，Java 直接中继其后续 token；若判断结果指向 Java 侧确定性能力（登录、推荐、匹配、分析），Runtime 在抛出 directive 后即按理解收口的方式停止，不再进入合成，由 Java 接管执行对应能力。

这一选型把“是否在流内继续合成”的开关，建立在能力卡的执行归属之上。为了不让 Runtime 核心硬编码 Boss 直聘、简历等业务概念，执行归属应作为 Profile 能力卡的一个声明式字段下沉到配置层，例如为每个能力声明 `executor: backend` 或 `executor: runtime`。Runtime 在路由完成后读取目标能力的 `executor`：标记为 `runtime` 的继续合成，标记为 `backend` 的抛出 directive 后收口。如此既消除了独立理解往返，又保持了 Runtime 核心的业务无关性。

### 3.2 目标链路

目标链路下，一轮托管问答只发起一次 Runtime 流式请求。事件顺序为：Runtime 先回传一个携带 directive 与意图的早期事件（暂命名 `directive`），Java 收到后据 `executor` 归属决策；对于 `executor: runtime` 的能力，Runtime 紧接着回传 `reasoning_delta` 与 `message_delta` 直至 `done`，Java 全程中继；对于 `executor: backend` 的能力，Runtime 在 `directive` 之后直接 `done`，Java 转入既有的 `handleDirective` 确定性分发。预判分类 `intent_precheck` 与安全门控保持不变，仍在发起 Runtime 请求之前完成，作为提示注入而非权威路由。

## 4. 详细设计

### 4.1 agent-runtime：流内早期 directive 事件

在流式 SSE 端点与 Agent 图的衔接处，新增一个在 Task Understanding 完成、进入合成之前抛出的早期事件。该事件至少包含 `domain`、`intent`、`confidence`、`risk`、`needs_clarification`、`next_action`、`slots` 与完整 directive，字段命名与现有 `IntentResult` 及 directive 结构保持一致，确保 Java 侧零成本解析。路由完成后，依据目标能力卡的 `executor` 字段决定后续路径：`runtime` 继续既有合成节点，`backend` 走与 `understanding_only` 等价的收口逻辑，仅是改为在流式通道内先发 `directive` 再发 `done`。`understanding_only` 作为显式短路入口可以保留，用于纯路由探测等场景，但不再是托管问答关键路径的必经环节。

### 4.2 agent-runtime：能力卡执行归属声明

在 Profile 能力卡 schema 中新增可选字段 `executor`，取值 `backend` 或 `runtime`，缺省按 `runtime` 处理以兼容存量配置。job-buddy Profile 中，将 Boss 登录、岗位推荐、简历匹配、简历分析四类由 Java 服务执行的能力标注为 `backend`，其余开放域问答、复杂工程问答等标注为 `runtime`。该字段仅作路由分流信号，不参与权限判定，权限仍由独立的权限服务负责。

### 4.3 agent-backend：单流中继与流内路由

`AgentIntegrationServiceImpl.runRuntimeStream` 的事件解析需要新增一类 `directive` 事件回调，与现有 `token`、`reasoning`、`done`、`error` 并列。`ChatSseServiceImpl` 改造为：托管问答路径不再先发 `runTaskUnderstanding` 非流式请求，而是直接发起一次流式托管请求；在 `directive` 回调里完成 `intent` 事件下发与能力分发判断。判断为 `backend` 归属时，主动结束当前流式中继并转入 `handleDirective`；判断为 `runtime` 归属时，继续把后续 token 中继给前端。个人上下文只在这一次请求里装配一次，彻底消除理解阶段与合成阶段的双重装配。

需要特别处理的是 `backend` 归属能力被识别后的流式收束：Runtime 在该情形下只发 `directive` 与 `done`，Java 收到 `directive` 即可中断等待并转入确定性分发，无需等待答案 token。为避免 Runtime 已部分执行的副作用被二次触发，确定性分发只覆盖 Java 自有服务的能力，不重复调用 Runtime。

### 4.4 SSE 事件契约变更

本方案新增一个 Runtime 流式 `directive` 事件，属于 SSE 主链路契约变更。需同步更新 Runtime 流式端点文档、Java 中继解析、前端对未知事件的容错（前端无需展示该事件，但需保证不报错）。`intent_precheck`、`intent`、`tool_status`、`message_delta`、`reasoning_delta`、`done`、`error` 等既有事件语义不变。

## 5. 涉及模块与接口

本方案触及对话主链路、意图路由、SSE 事件契约与 Profile 能力卡四个核心面，涉及模块包括 agent-runtime 的 Agent 图与流式端点、Profile 能力卡定义与加载、agent-backend 的 `ChatSseServiceImpl` 与 `AgentIntegrationServiceImpl`、agent-frontend 的 SSE 事件容错，以及 agent-eval 的路由评分与用例、`.agent-harness` 的相关门禁。按 CLAUDE.md 的 Harness 与 Eval 联动规则，意图分类输出、SSE 事件、能力路由的变更必须同步检查并更新 `agent-eval/app/grader.py`、`agent-eval/cases/`、`agent-eval/tests/` 与 `.agent-harness/scripts/evaluate.sh`。

## 6. 风险与对策

第一类风险是路由正确性。理解从独立往返改为流内早期事件后，若 `executor` 标注遗漏或错误，会导致本应由 Java 执行的确定性能力被 Runtime 误合成，或反之。对策是为四类 backend 能力补充契约测试，断言其 `directive` 后即收口且不产生 `message_delta`，并在 eval 路由用例中覆盖每类能力的归属判定。

第二类风险是流式中断语义。`backend` 归属能力在流内被识别后需要主动中断中继，若中断处理不当可能造成连接悬挂或前端 loading 不收尾。对策是复用现有 `done`/`error` 收尾路径，确保任意分支都以明确终态事件结束，并在前端验证 loading 态必然结束。

第三类风险是契约兼容性。新增 `directive` 事件若前端未做未知事件容错，可能触发解析异常。对策是前端对未识别事件静默忽略，并在灰度阶段保留 `understanding_only` 旧路径作为回退开关，通过配置可一键切回双往返链路。

第四类风险是不可在本环境完整验证。该改动属于核心链路与 SSE 契约变更，必须经过 agent-eval 路由回归与真实工作台浏览器端到端验证（首字等待缩短、答案逐字渲染、四类确定性能力分发正确、loading 收尾正常）。在缺少真实模型与 Boss 登录态的环境中无法完成端到端确认，需在交付说明中明确标注未覆盖项。

## 7. 验证方案

代码层面，agent-runtime 补充单测覆盖流内早期 `directive` 事件的字段完整性、`executor` 分流逻辑与 `backend` 归属下的流式收口；agent-backend 补充 `runRuntimeStream` 对 `directive` 事件的解析与分发契约测试；agent-eval 更新路由评分用例覆盖四类 backend 能力与 runtime 能力的归属判定。门禁层面，按模块运行 `.agent-harness/scripts/gate.sh agent-runtime --quick`、`gate.sh agent-backend --quick`、`gate.sh agent-eval --quick`。体验层面，在真实工作台分别触发开放域问答、复杂工程问答、Boss 登录、岗位推荐、简历匹配、简历分析六条路径，确认托管问答只发起一次 Runtime 请求、首字等待较双往返链路明显缩短、四类确定性能力分发与现状一致、各路径 loading 均正常收尾。

## 8. 后续演进

单流路由落地后，首字关键路径上仅剩合成 prefill 与新会话远程库初始化两段耗时。合成 prefill 可结合 Prompt 缓存命中率优化与上下文装配进一步压缩；新会话远程库初始化可通过会话预热或惰性初始化在首包之后异步完成。理解阶段产出的中间状态（如已抽取槽位、候选能力评分）在单流内得以传递给合成阶段，为后续“理解结果直接驱动合成上下文裁剪”提供了基础，可进一步减少合成阶段的重复装配与召回。

## 9. 落地状态

本文档为方案设计稿，对应的 Runtime 流内 `directive` 事件、能力卡 `executor` 字段、Java 单流中继改造与 eval 用例更新尚未实现。由于该改动涉及核心链路与 SSE 契约，且无法在当前环境完成 eval 回归与浏览器端到端验证，代码落地需在确认设计后另行排期，并以本文档为实现契约。同批次已先行落地的、与本方案不冲突的安全优化包括：agent-backend RestTemplate 接入 Apache HttpClient 连接池、托管问答空产出回退路径复用已装配元数据以消除一次重复的个人上下文装配、agent-runtime LLM 客户端复用 httpx 连接、Trace 落盘改为线程池异步以避免阻塞事件循环。
