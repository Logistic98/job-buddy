# 简历互导联动与岗位选中分析方案

## 一、为什么做

当前工作台存在岗位分析体验断层：岗位推荐结果只能整体追问，用户无法选中某一个岗位做定向分析；收藏的岗位也没有沉淀分析结论，每次都要重新提问，分析结果不可回看、不可比较。

原计划同时补齐“简历管理与简历撰写互相导入联动”，但当前实现方向与预期偏差较大，先暂停该链路。现阶段简历撰写页仅维护页面本地 Markdown 草稿，不与简历管理 `ResumeRecord` 交互，也不提供 Markdown 简历落库接口。后续重新设计清楚数据边界后再恢复联动。

## 二、方案是什么

### 2.1 简历撰写与简历管理边界

现阶段简历撰写页只负责浏览器本地草稿编辑、预览和导出，保存按钮仅更新本页面 localStorage 草稿状态，不写入简历管理，不创建 `ResumeRecord`，不影响当前会话简历选择。简历管理仍只负责已上传、已解析或画像类简历。两者后续是否互通，需要重新设计 Markdown 简历的数据模型、对象存储、预览下载、解析状态和匹配上下文边界。

### 2.2 岗位选中分析

- 对话入口：聊天岗位卡片新增"分析此岗位"按钮，点击后把该岗位结构化信息作为 `selectedJob` 附带在 `/api/chat/stream` 请求中，后端在转发 Runtime 时将选中岗位注入任务上下文，回答按常规问答消息持久化到会话，无需新增存储。
- 收藏入口：收藏表 `job_buddy_job_favorite` 新增 `analysis_json`、`analyzed_at` 两列；新增 `POST /api/jobs/favorites/{jobKey}/analyze` 接口，后端取默认简历解析文本与收藏岗位 JSON，经既有 `RuntimeToolClient` 调用 Runtime `resume_match` 工具，结果写回收藏行。收藏页展示分析得分、建议与分析时间，并提供"重新分析"按钮覆盖旧结果。

## 三、具体怎么做

### 3.1 后端（agent-backend）

1. 简历撰写页不提供 `POST /api/resume/markdown` 后端落库接口，页面草稿仅保存在浏览器本地。
2. Flyway 新增 `V1_11_0__Add_job_favorite_analysis.sql`：`ALTER TABLE job_buddy_job_favorite ADD COLUMN analysis_json TEXT NULL, ADD COLUMN analyzed_at DATETIME NULL;` 实体与查询同步增加两字段。
3. `JobFavoriteController` 新增 `POST /api/jobs/favorites/{jobKey}/analyze`，可选请求体 `{resumeId}`；Service 中取收藏岗位 JSON 与简历 parsed 文本，调用 `RuntimeToolClient.invoke("resume_match", {resume, jobs:[job], top_k:1})`，把首条 match 与简历标识组装为 analysis JSON 落库并返回。Runtime 不可达或工具失败时返回明确错误信息，不写库。
4. `ChatStreamRequest` 新增 `selectedJob`（Map）；`ChatSseServiceImpl` / `AgentIntegrationServiceImpl` 在构造 Runtime 请求时，若存在 selectedJob 则把岗位关键信息拼入用户任务上下文（不改 Runtime 接口契约，走 message 上下文注入），其余持久化逻辑复用现状。

### 3.2 前端（agent-frontend）

1. `ResumeWriter.vue`：保存按钮仅更新本页面本地草稿；不调用 resume store，不写入简历管理，不提供“从简历管理导入”入口。
2. `ResumeManager.vue`：不为 Markdown 草稿提供“在撰写中打开”入口。
3. `ChatPanel.vue` 岗位卡片新增"分析此岗位"按钮，点击后通过 chat store 以 selectedJob 发起一次流式问答。
4. `JobCardList.vue` 收藏模式：每张卡片显示分析摘要（得分、建议、analyzed_at），提供"分析 / 重新分析"按钮，调用收藏分析接口并就地刷新。
5. `src/api/jobs.js`、`src/stores/job.js` 增加岗位分析封装；`src/api/resume.js`、`src/stores/resume.js` 仅保留简历管理相关 API，不承载 Markdown 草稿保存。

## 四、涉及模块与接口

| 模块 | 变更 |
| --- | --- |
| agent-backend | 新增 POST /api/jobs/favorites/{jobKey}/analyze；ChatStreamRequest 增加 selectedJob；Flyway V1_11_0 |
| agent-frontend | ResumeWriter 仅本地保存；ChatPanel / JobCardList 岗位分析交互；api 与 store 封装 |
| agent-runtime | 无接口变更，复用 resume_match 工具与 /v1/runtime/tools/{name}/invoke |

## 五、风险与注意点

- 简历撰写与简历管理暂不互通，避免错误的 Markdown 简历记录污染简历库、当前会话简历和岗位匹配上下文。
- 收藏分析是同步调用 Runtime LLM 工具，耗时可能较长，前端需要 loading 态与超时提示；后端沿用 RestTemplate 既有超时配置。
- selectedJob 注入上下文时只取岗位名、公司、薪资、要求等关键字段，控制 token 体积，防止把整段 job_json 灌入。
- jobKey 来自前端 favoriteKey 规则（securityId→id→jobId→组合名），分析接口必须按 (user_id, job_key) 定位，避免跨用户访问。
- 重新分析直接覆盖 analysis_json 与 analyzed_at，不保留历史版本（首期约束，后续可演进为分析历史表）。

## 六、如何验证

- 后端：新增 Controller/Service 单测覆盖参数校验、空简历、Runtime 失败分支；执行 `./.agent-harness/scripts/verify.sh agent-backend --quick`。
- 前端：`npm run build` 通过后启动本地三件套，浏览器验证：撰写页点击保存只更新本地已保存状态且不发起后端草稿保存请求；聊天岗位卡片点击分析→流式回答并持久化；收藏页分析→显示得分与时间→重新分析时间刷新→刷新页面结果仍在。
- 门禁：`./.agent-harness/scripts/gate.sh agent-backend --quick` 与 frontend 对应检查。

## 七、后续演进

- 收藏分析支持选择指定简历版本、保留多次分析历史与对比视图。
- 重新设计简历撰写与简历管理互通方案，明确 Markdown 简历是否应进入 `ResumeRecord`、如何生成可预览文件、如何参与匹配与版本管理。
- selectedJob 升级为 Runtime 显式接口字段，纳入 Trace 与 Eval 用例。
