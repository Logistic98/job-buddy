## Tsenta 竞品调研报告

调研对象：Tsenta  
官网：https://tsenta.com/  
第三方资料：https://moge.ai/zh/product/tsenta#overview、https://navtools.ai/tool/tsenta

## 一、产品概述

Tsenta 是一款面向求职者的 AI 求职自动化产品，核心定位是“自动发现匹配岗位、为每个岗位定制简历与求职信，并自动提交申请”。其公开页面强调它由 Y Combinator 支持，主打面向高频求职用户的端到端自动化能力。与传统只做职位聚合、简历优化或表单自动填充的工具不同，Tsenta 试图把求职链路中的“找岗位、改材料、填表、提交、追踪反馈”整合为一个连续的 Agent 工作流。

从官网信息看，Tsenta 的最新产品形态已经不再只是 Moge 页面中描述的本地桌面应用，而是同时提供 Dashboard、iMessage、WhatsApp、Chrome Extension、Claude/Codex MCP 等多个入口。它强调监控 50,000+ 公司招聘页面，覆盖 Workday、Greenhouse、Lever、Ashby 等 10 多类 ATS，并在匹配岗位出现后快速触发申请流程。产品承诺用户可以先免费使用 25 次申请，后续按月订阅并按申请量分层收费。

## 二、核心功能拆解

### 1. 岗位发现与实时监控

Tsenta 的第一阶段能力是“Find”。产品宣称会监控 50,000+ 公司职业页面，当有符合用户简历与偏好的新岗位上线时，会在匹配流中实时出现，并给出匹配原因。它强调“先发优势”，即尽量让用户成为前 100 个申请者之一。这个定位直接命中海外求职市场中竞争激烈、岗位很快满员的问题。

这一点与普通岗位搜索工具不同。普通工具多以用户主动搜索为主，而 Tsenta 更偏向后台持续监控和事件触发。对用户而言，它把“每天刷职位”变成“岗位主动找我”。

### 2. 简历与求职信针对性改写

Tsenta 的第二阶段是“Prep”。它会基于具体岗位描述读取关键词、技能、招聘方关注点，并为每个岗位生成 ATS 友好、招聘者可读的定制简历和求职信。官网示例展示了简历条目的改写 diff，例如把普通描述改写为更量化、更结果导向的表达。

值得注意的是，Tsenta 强调“每个修改都会展示给用户，未经确认不会静默发送”。这说明它在自动化与可控性之间做了平衡：AI 负责生成与改写，用户保留审核与编辑权。

### 3. 自动申请与表单填写

Tsenta 的第三阶段是“Apply”。它覆盖 Workday、Greenhouse、Lever、Ashby 等主流 ATS，能够处理登录、字段填写、开放式问题回答、简历上传和最终提交。官网强调用户可以观看整个过程，也可以选择不看。

Moge 与 NavTools 的资料也指出，Tsenta 可以端到端处理完整申请流程，并且支持复杂多页表单和开放问题。这是它相对简历工具或职位搜索工具的关键差异：它不是只生成材料，而是进入最终申请动作。

### 4. 申请回执与流程追踪

Tsenta 的第四阶段是“Track”。它会将招聘方邮件回复自动路由回对应申请记录，并自动更新申请状态，例如 Applied、Viewed、Replied、Interview 等。官网还展示了每次提交后的 receipt，包括填写字段、开放题回答、上传的简历和 ATS 确认信息。

这类“申请回执”设计很重要，因为自动申请最容易引发用户不信任：到底投了什么、有没有乱填、有没有漏字段。Tsenta 通过 receipt 和状态追踪降低黑盒感。

### 5. 多入口 Agent 化

Tsenta 官网展示了多个入口：Web Dashboard、iMessage、WhatsApp、Chrome Extension、Claude/Codex MCP。尤其是 MCP 入口，说明其产品思路已经从单一应用扩展到“求职 Agent 能力作为外部工具被调用”。用户可以在 Claude Code/Codex 中触发“apply to the new Stripe role”之类的命令。

这对 job-buddy 有参考价值：求职能力不一定只存在于前端页面，也可以被封装为工具、MCP 服务、浏览器扩展或消息入口。

## 三、商业模式与指标

根据官网，Tsenta 当前定价按申请量分层，而不是按功能分层。所有套餐都包含完整产品能力，区别主要是每 30 天可提交的申请数量。

- 免费：前 25 次申请免费，无需信用卡。
- Starter：19 美元/月，600 次申请/30 天。
- Pro：39 美元/月，1,500 次申请/30 天。
- Power：99 美元/月，4,500 次申请/30 天。

第三方工具站展示其月访问量约 55.4K，主要流量来自美国、印度、英国、加拿大、德国等地区。Moge 页面显示月访问量约 55.39K，平均访问时长约 19 秒。不同站点数据口径可能不同，但可以判断 Tsenta 已经具备一定曝光度，仍处于增长和早期验证阶段。

## 四、Tsenta 的优势

### 1. 闭环完整

Tsenta 最大优势是覆盖求职完整闭环：岗位发现、材料优化、自动投递、状态追踪。很多竞品只覆盖其中一个环节，例如简历优化、职位推荐或浏览器自动填表。Tsenta 的价值主张更强，因为它直接减少用户每天重复操作的时间成本。

### 2. 自动化程度高

它不是“辅助用户填几个字段”，而是明确承诺打开 ATS 表单并完成提交。对大量投递的用户，尤其是学生、应届生、OPT/H-1B 等需要高频申请的人群，这种自动化强度具有明显吸引力。

### 3. 强调透明与可控

Tsenta 反复强调申请过程可见、修改可审查、提交有回执、用户可暂停和编辑。这说明它意识到自动投递产品的核心信任问题。相比“后台偷偷帮你投”，这种设计更容易被用户接受。

### 4. 数据入口广

它既监控公司官网，也支持用户粘贴外部职位 URL，还提供浏览器扩展和消息入口。这使得用户不必完全迁移到一个新平台，而可以在已有工作流中调用 Tsenta。

### 5. 结果导向明确

官网展示“用户最终入职的公司 Logo”，以及 applicant 排名、提交数量、回复状态等指标，强化了“我能帮你更快拿面试”的结果感。

## 五、潜在弱点与风险

### 1. 自动投递合规与平台风控风险

自动化申请涉及 ATS、招聘网站和公司官网的使用规则。过高频率的自动提交可能触发反机器人、账号限制或平台封禁。即使 Tsenta 强调“像普通申请一样提交”，长期看仍存在合规和风控压力。

### 2. 质量与数量的矛盾

按申请量收费容易鼓励高频投递，但高质量求职并不只是海投。对于中高级岗位，过度自动化可能导致岗位理解不足、材料个性化不足、投递策略失真。Tsenta 需要持续证明其定制材料不是模板化改写。

### 3. 本地隐私与云端多入口之间的张力

Moge/NavTools 信息强调本地设备执行，但官网同时展示 Dashboard、消息应用、MCP 等入口。这意味着其架构可能已经从纯本地扩展为混合形态。如何解释数据在哪里处理、凭证如何保存、哪些动作在云端完成，是它需要持续澄清的问题。

### 4. 国际化与本地招聘生态适配

Tsenta 当前主要面向海外 ATS 生态，如 Workday、Greenhouse、Lever、Ashby。对中国市场的 Boss 直聘、猎聘、拉勾、脉脉、校招系统等，本地化适配难度较大。job-buddy 如果定位中国求职场景，反而有机会做深本土化。

## 六、与 job-buddy 现有能力对比

当前 job-buddy 已经具备较多求职链路能力，包括工作台对话、Boss 直聘扫码登录与岗位搜索、求职画像、岗位收藏、求职历程、简历管理、简历撰写、简历分析、笔试题库、笔试模拟、平台设置、记忆管理、黑名单、服务健康状态等。这些能力说明 job-buddy 已经不只是简历工具，而是一个面向求职全过程的本地化 Agent 工作台。

与 Tsenta 相比，job-buddy 的优势在于更贴近中文求职场景，已经围绕 Boss 直聘、岗位筛选、简历匹配、求职画像、笔试准备和求职历程建立了完整模块。它更像“个人求职操作系统”，而 Tsenta 更像“自动投递 Agent”。

但差距也比较明显：Tsenta 更强调自动发现、自动投递、申请回执和多入口调用；job-buddy 当前更偏用户主动发起搜索、管理和辅助决策，自动化闭环还没有完全打通。

## 七、对 job-buddy 的功能完善建议

### 1. 建设“岗位监控与自动发现”能力

建议在现有 Boss 岗位搜索基础上增加岗位监控任务。用户可以保存一组求职条件，例如城市、岗位关键词、薪资、经验、排除词、公司黑名单、行业偏好。系统定时检索并将新增岗位写入“岗位发现池”，标记为新增、已读、已收藏、已忽略。

这会把当前“用户问一次、系统搜一次”的模式升级为“系统持续监控、用户集中处理”的模式。对求职者而言，这比单次推荐更有持续价值。

### 2. 增加岗位匹配解释与排序理由

Tsenta 强调岗位匹配百分比和匹配原因。job-buddy 也应该在岗位卡片中展示明确的匹配解释，例如技能匹配、经验匹配、薪资匹配、城市匹配、风险项、黑名单命中情况。建议把匹配分拆成多个维度，而不是只有一个总体评分。

可以设计为：综合匹配分、技能契合度、履历契合度、薪资契合度、稳定性风险、外包/驻场风险、通勤/城市匹配、成长性判断。这样用户能快速判断为什么推荐或为什么不推荐。

### 3. 简历针对岗位的差异化改写

当前 job-buddy 已有简历管理、简历撰写和简历分析模块，建议进一步加入“针对某个岗位生成定制版简历”的能力。用户在岗位收藏或推荐结果中点击“定制简历”，系统读取岗位 JD、用户求职画像、历史简历，生成一版专门面向该岗位的简历，并以 diff 方式展示修改点。

Tsenta 的简历改写示例非常值得参考：不只生成完整简历，还要展示“原句 -> 新句”的变化，并解释为什么这样改。这样能增强用户信任，也方便用户学习如何优化表达。

### 4. 建立“申请包”概念

建议为每个岗位生成一个申请包，包含岗位信息、匹配分析、定制简历、求职信/沟通话术、开放问题回答、笔试准备清单、投递状态和后续跟进记录。这样岗位收藏不只是收藏列表，而是进入一个完整的求职任务单。

申请包可以与求职历程打通：当用户把岗位推进到“已投递、笔试、一面、二面、Offer、结束”时，系统自动归档每一阶段材料和记录。

### 5. 增加申请回执与操作审计

Tsenta 的 receipt 设计值得借鉴。job-buddy 如果后续要做自动投递或半自动投递，必须记录每次操作：投递平台、填写字段、上传文件、回答内容、提交时间、是否成功、失败原因、截图或 HTML 摘要。即使短期不做自动提交，也可以先做“人工投递回执”，让用户手动记录投递结果。

这会显著提升求职历程模块的价值，使其从普通表格变成可追溯的求职证据链。

### 6. 自动投递应先做“半自动确认模式”

考虑到 Boss 直聘和国内招聘平台风控较强，不建议一开始就做完全自动投递、自动打招呼或代聊。更稳妥的路径是半自动确认模式：系统自动准备材料、打开目标页面、辅助填充或生成话术，但最终点击发送或提交由用户确认。

这与项目当前“安全边界”一致：不自动投递、不自动打招呼、不代聊。建议继续坚持这个边界，优先做“高质量辅助”和“可控自动化”。

### 7. 强化求职画像的数据结构

求职画像已经存在，但建议进一步结构化到能驱动推荐和自动回答问题的程度。例如工作授权、期望城市、远程偏好、薪资底线、行业偏好、公司规模偏好、可接受通勤、排除项、当前状态、到岗时间、是否接受外包、是否接受驻场、是否接受大小周等。

这些字段不只是展示信息，而应该被岗位筛选、简历改写、开放问题回答、匹配解释共用。

### 8. 扩展消息入口或轻量通知

Tsenta 提供 iMessage 和 WhatsApp 入口，核心价值是“有新岗位时及时触达用户”。job-buddy 可以先做轻量版本，例如本地桌面通知、邮件通知、企业微信/Telegram 通知，提醒用户“新增 12 个高匹配岗位，其中 3 个建议优先处理”。

通知不需要一开始做复杂交互，只要把用户拉回工作台即可。

### 9. 打通笔试题库与岗位 JD

当前 job-buddy 已经拆分为笔试题库和笔试模拟。建议下一步把岗位 JD 与笔试模拟关联起来：用户收藏某个 Java 后端岗位后，系统自动识别岗位技术栈，例如 Java、Spring、MySQL、Redis、Kafka、分布式、Agent 工程，然后从题库中生成“该岗位专项笔试模拟”。

这会让笔试模块从孤立练习变成岗位驱动的准备工具，差异化更强。

### 10. 建设求职漏斗分析

Tsenta 的 Track 阶段展示 Applied、Viewed、Replied、Interview 等状态。job-buddy 已有求职历程，可以进一步做统计分析：投递数量、回复率、笔试率、面试率、Offer 率、不同岗位方向转化率、不同简历版本转化率、不同平台转化率。

这类漏斗分析能帮助用户优化求职策略，而不是只堆积岗位记录。

## 八、建议优先级路线图

短期建议优先完善三个方向。第一，岗位匹配解释和申请包，把当前岗位收藏、简历、求职历程串起来。第二，岗位 JD 驱动的定制简历和笔试模拟，让已有简历与题库模块产生联动。第三，求职画像字段结构化与下拉框规范化，保证推荐、匹配、简历生成和话术生成都有稳定输入。

中期可以建设岗位监控和通知机制，让系统主动发现新岗位，并把新增岗位推送到工作台。随后再做申请回执、投递记录和求职漏斗分析。这样 job-buddy 会从“求职助手”升级为“求职运营系统”。

长期如果要对标 Tsenta 的自动申请能力，建议优先做浏览器辅助式半自动申请，而不是全自动提交。系统可以打开页面、识别字段、生成填写建议和上传材料，但关键提交动作由用户确认。这样既保留效率，也降低账号风控和合规风险。

## 九、结论

Tsenta 的核心竞争力不是单点 AI 能力，而是把求职链路做成了连续 Agent 工作流，并用实时监控、定制材料、自动申请、状态追踪构成闭环。它给 job-buddy 的最大启发是：求职产品的价值不只在“生成内容”，而在“持续推进用户拿到面试和 Offer 的全过程”。

job-buddy 目前已经具备中文求职场景下的多个关键模块，尤其是 Boss 直聘接入、求职画像、岗位收藏、求职历程、简历管理与笔试模块，基础比单一工具更完整。下一步应重点加强模块之间的联动，把“岗位 -> 匹配 -> 定制简历 -> 笔试准备 -> 投递记录 -> 跟进分析”串成闭环，并在安全可控的前提下逐步加入半自动化能力。
